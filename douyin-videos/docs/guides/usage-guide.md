# 智能体优化模块使用说明

**版本**: 1.0
**创建日期**: 2026-04-11
**适用范围**: 抖音视频采集与分析系统

---

## 一、概述

### 1.1 模块组成

本系统包含三个独立的优化模块：

| 模块 | 文档路径 | 用途 | 发送时机 |
|------|----------|------|----------|
| **问题记录** | `docs/issue-logging-design.md` | 记录运行中的异常情况 | **运行时发送** |
| **关键词优化** | `docs/keyword-optimization-design.md` | 优化关键词列表，提高搜索效果 | **运行后发送** |
| **博主优化** | `docs/blogger-optimization-design.md` | 优化博主质量，屏蔽低质量博主 | **运行后发送** |

### 1.2 核心原则

```
运行时：伴随式记录异常
运行后：基于验收标准进行优化
```

        │ → 综合判断（持续跟进后）     │

        │ → 检查验收标准               │
        │   ├─ 通过 → 完成             │

## 三、问题判断规则（核心）

### 3.1 判断流程图

```
模块运行完成后
    │
    ├──→ 是否运行报错？
    │     ├──→ 是 → ✅ 记录问题
    │     └──→ 否 ↓
    │
    └──→ 验收标准是否通过？
          ├──→ 是 → ❌ 不记录（正常完成）
          └──→ 否 → ✅ 记录问题（验收未通过）
```

### 3.2 问题类型速查表

| 触发场景 | 问题类型 | type值 | 优先级 |
|----------|----------|--------|--------|
| 视频分析失败 | 分析失败 | analysis_failed | medium |
| 数据文件损坏 | 数据异常 | data_error | high |
| 流程执行错误 | 流程异常 | process_error | high |
| API调用报错 | API异常 | api_error | high |
| 配置文件错误 | 配置异常 | config_error | high |
| 验收标准未通过 | 验收未通过 | acceptance_failed | medium |

### 3.3 边界场景示例

| 场景 | 是否记录 | 判断依据 |
|------|----------|----------|
| 视频分析完全失败 | ✅ 记录 | 运行报错 |
| 视频分析部分成功但影响后续 | ✅ 记录 | 等同于运行失败 |
| 关键词优化后质量提升≥10% | ❌ 不记录 | 验收通过 |
| 关键词优化后质量提升<10% | ✅ 记录 | 验收未通过 |
| 博主优化后黑名单比例≤20% | ❌ 不记录 | 验收通过 |
| 博主优化后黑名单比例>20% | ✅ 记录 | 验收未通过 |
| API返回429错误 | ✅ 记录 | 运行报错 |
| 用户主动取消 | ❌ 不记录 | 用户行为 |
| 黑名单博主被跳过 | ❌ 不记录 | 正常业务逻辑 |

## 四、输出文件说明

### 4.1 问题记录文件

**文件路径**: `d:\opencli\douyin-videos\optimization-feedback.json`

**用途**：存储所有运行中发现的异常和验收未通过的问题

**查看方式**：
```bash
cat optimization-feedback.json | jq '.issues[]'
```

### 4.2 关键词优化报告

**文件路径**: `d:\opencli\douyin-videos\reports\keyword-optimization-{日期}.md`

**用途**：关键词优化的详细报告，包含评估结果和建议

### 4.3 博主优化报告

**文件路径**: `d:\opencli\douyin-videos\reports\blogger-optimization-{日期}.md`

**用途**：博主优化的详细报告，包含质量评分和处理建议

## 五、快速启动指南

### 5.1 第一次使用

```bash
# 1. 确保数据目录存在
cd d:\opencli\douyin-videos

# 2. 检查是否有历史问题记录

```

      ↓ （伴随发送 issue-logging-design.md）

步骤2：阶段三完成后，发送 keyword-optimization-design.md
      ↓

步骤3：关键词优化完成后，发送 blogger-optimization-design.md
      ↓

```bash
# 查看所有待处理问题
cat optimization-feedback.json | jq '.issues[] | select(.status=="pending")'

# 查看高优先级问题
cat optimization-feedback.json | jq '.issues[] | select(.priority=="high")'

# 查看某类问题
cat optimization-feedback.json | jq '.issues[] | select(.type=="analysis_failed")'
```

## 六、常见问题

### Q1: 什么时候需要发送问题记录文档？

**A**: 在执行阶段一、二、三之前或同时发送。智能体会在整个运行过程中监控并记录异常。

### Q2: 关键词优化和博主优化的顺序可以调换吗？

**A**: 可以。两者相互独立，但建议先进行关键词优化（因为博主优化可能依赖关键词信息）。

### Q3: 如果验收标准一直不通过怎么办？

**A**: 查看 optimization-feedback.json 中的 acceptance_failed 类型问题，根据具体未通过的指标进行调整。

### Q4: 问题记录文件会无限增长吗？

**A**: 不会。当问题数量超过1000个时，会自动清理已解决的问题。

### Q5: 可以手动标记问题为已解决吗？

**A**: 可以。修改 optimization-feedback.json 中对应问题的 status 字段为 "resolved"。
